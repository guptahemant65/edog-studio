# F27 P10 — FLT Phase 0 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the FLT-side DevMode dispatch surface (5 seam interfaces + 3 controllers + security middleware + capability endpoint + G7 hardening + contract-compatibility tests) that EDOG Phase 1 will consume. Tag `flt/qa-contract-v1.0` on merge.

**Architecture:** All new FLT code lives under `Service\Microsoft.LiveTable.Service\DevMode\Qa\` (controllers, seams, registration helpers, middleware) and `Service\Microsoft.LiveTable.Service\DevMode\Attributes\` (marker attributes), guarded by `#if DEVMODE`. FLT Phase 0 exposes the types and the opt-in registration path only; `WorkloadEndpointSetup.AddAllControllersFromAssembly` must not surface the QA controllers by default, and EDOG Phase 1 will call the opt-in registration extension later. The five seams wrap real FLT abstractions where they already exist (`LiveTableHandler` / scheduler status flow, `IFileSystemFactory`, timer topic) and use explicit late-bound bridges or test doubles where EDOG owns the runtime hookup.

**Tech Stack:** ASP.NET Core, C#, MSTest, Moq, Swashbuckle, Microsoft.Extensions.DependencyInjection, Microsoft.PowerBI.ServicePlatform.WireUp.

---

## File Structure

`Service\Microsoft.LiveTable.Service\Microsoft.LiveTable.Service.csproj` gains explicit Debug/Release `DefineConstants` groups so `DEVMODE` exists only in Debug builds of the service assembly. This file is the hard compile-time gate for all QA types.

`Service\Microsoft.LiveTable.Service.EntryPoint\Microsoft.LiveTable.Service.EntryPoint.csproj` mirrors the `DEVMODE` symbol in the Debug host build so the EntryPoint can compile references to the gated service assembly without mismatched symbols.

`test\Microsoft.LiveTable.Service.UnitTests\Microsoft.LiveTable.Service.UnitTests.csproj` adds `DEVMODE` in Debug so unit tests can compile against the gated QA types, while Release test builds stay clean and verify absence.

`test\Microsoft.LiveTable.Service.FeatureTests\Microsoft.LiveTable.Service.FeatureTests.csproj` adds `DEVMODE` in Debug so host-based smoke tests can exercise the gated controllers when explicitly enabled.

`Workload.LiveTable.sln` is updated once, at the compatibility-test milestone, to include the new `Microsoft.LiveTable.Service.QaContractCompat.Tests` project so the standard solution build covers the new gate.

`Service\Microsoft.LiveTable.Service\Initialization\WorkloadEndpointSetup.cs` is the only core runtime file that changes in Phase 0. It installs the filtered controller activator so QA controllers are never activated by the default auto-discovery path, even in Debug builds.

`Service\Microsoft.LiveTable.Service\DevMode\Attributes\EdogDirectInvokeSeamAttribute.cs` defines the direct-invoke opt-in marker used by `IQaDirectInvokeRegistry` to enumerate only explicitly tagged methods.

`Service\Microsoft.LiveTable.Service\DevMode\Attributes\EdogFileEventSeamAttribute.cs` defines the file-event opt-in marker used by future catalog scanners and unit tests that verify the seam shape.

`Service\Microsoft.LiveTable.Service\DevMode\Attributes\EdogTimerSeamAttribute.cs` defines the timer opt-in marker used by future catalog scanners and timer-injection tests.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\QaDispatchContracts.cs` centralises the contract DTOs and enums used by the controllers, seams, middleware, and compatibility tests: request, accepted response, poll response, capability response, captured topics, and verdict/kind enums.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\IQaDirectInvokeRegistry.cs` contains the direct-invoke seam contract and its slot/result record types. It is the source of truth for DI slot enumeration and invocation.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\QaDirectInvokeRegistry.cs` reflects over explicitly supplied seam targets, builds the slot table from `[EdogDirectInvokeSeam]`, binds `JToken` parameters into method arguments, and invokes the target instance with cancellation support.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\IQaSignalRObserver.cs` defines the observer contract for register/harvest/dispose around correlation IDs and channel keys.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\QaSignalRObserver.cs` is the in-process implementation that buffers observed payloads per correlation ID and returns them as `QaCaptureEnvelope` instances.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\IQaDagRunInvoker.cs` defines the async DAG seam contract: trigger, poll, and cancel, plus the state record used by `QaAsyncDispatchController`.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\QaDagRunInvoker.cs` wraps `LiveTableHandler`’s existing run/status/cancel logic and tracks dispatch IDs back to workspace, artifact, and iteration IDs.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\IQaFileEventInjector.cs` defines the file-event seam contract that the sync dispatch controller uses.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\QaFileEventInjector.cs` late-binds to an internal publisher bridge so EDOG can connect the existing `IFileSystemFactory` DevMode wrapper without FLT registering anything by default.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\IQaTimerInjector.cs` defines the timer seam contract for synthetic tick injection.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\QaTimerInjector.cs` late-binds to an internal timer-topic bridge so EDOG can connect the existing timer topic without changing FLT’s default startup path.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\QaAuditLogWriter.cs` owns redaction, truncation, hash generation, and the final audit-log emission shape used by `QaSecurityMiddleware` and the controllers.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\QaSecurityMiddleware.cs` enforces `X-EDOG-Control-Token`, localhost-only access, 64 KB body cap, per-kind concurrency semaphores, and audit logging before any controller code runs.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\QaBuildInfoProvider.cs` surfaces `contractVersion`, `fltBuildSha`, and `schemaCapVersion` in one place so the capability endpoint and compatibility tests never duplicate the version strings.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\QaCapabilitiesController.cs` is the hidden DevMode-only capability route returning the version-skew contract expected by EDOG Phase 1 and G9.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\QaDispatchController.cs` is the hidden synchronous dispatch route that handles `DiInvocation`, `SignalRBroadcast`, `FileEvent`, and `TimerTick` only.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\QaAsyncDispatchController.cs` is the hidden asynchronous DAG-only route family for trigger, poll, and cancel.

`Service\Microsoft.LiveTable.Service\DevMode\Qa\QaControllerActivationGate.cs` is the single state holder that keeps QA controllers disabled unless EDOG explicitly turns them on through the late-registration path.

namespace Microsoft.LiveTable.Service.DevMode.Qa
{
    using System.Collections.Concurrent;
    using System.Threading;
    using System.Threading.Tasks;

    public sealed class QaSignalRObserver : IQaSignalRObserver
    {
        private readonly ConcurrentDictionary<string, QaCaptureEnvelope> captures = new();
        private readonly ConcurrentDictionary<string, string> channels = new();

        public void Register(string correlationId, string channelKey)
        {
            this.channels[correlationId] = channelKey;
            this.captures[correlationId] = new QaCaptureEnvelope();
        }

        public void Record(string correlationId, string channelKey, QaCapturedTopic topic)
        {
            if (!this.channels.TryGetValue(correlationId, out var expectedChannel) || expectedChannel != channelKey)
            {
                return;
            }

            this.captures[correlationId].Topics.Add(topic);
        }

        public Task<QaCaptureEnvelope> HarvestAsync(string correlationId, CancellationToken cancellationToken)
        {
            this.captures.TryGetValue(correlationId, out var captureEnvelope);
            return Task.FromResult(captureEnvelope ?? new QaCaptureEnvelope());
        }

        public void Dispose(string correlationId)
        {
            this.channels.TryRemove(correlationId, out _);
            this.captures.TryRemove(correlationId, out _);
        }
    }
}
#endif
