# FLT Mental Model — Always-Loaded Reference

This document is baked into the skill's context on every run. It covers the seven concepts the skill needs to reason over FLT without reading any FLT source code upfront.

---

## 1. Ports and Process Topology

```
Copilot CLI skill
      │  HTTP   localhost:5555
      ▼
EDOG dev-server  (scripts/dev-server.py)
      │  in-process interceptors + HTTP proxy
      ▼
FLT service      localhost:5557
```

▸ **5555** — EDOG dev-server. All skill tool calls go here.  
▸ **5557** — FLT service. The skill never calls this directly; EDOG proxies it.  
▸ **Headless start:** launch `scripts/dev-server.py` directly. It does NOT open a browser. The default `python edog.py` opens the EDOG Studio webpage — the skill must NOT use it.

## 2. The DAG and runDAG

FLT executes data refresh as a **DAG (Directed Acyclic Graph)** of nodes. Each node corresponds to a Materialized Lake View (MLV) and its upstream dependencies. The scheduler (`liveTableSchedule`) drives execution node-by-node, respecting dependency order.

**Triggering a DAG run:**

```
POST /api/flt-proxy/liveTableSchedule/runDAG/{iterationId}
```

◆ `iterationId` is a **fresh GUID generated by the skill** for each run. It is the operation ID (OpId) that flows through every interceptor event for that run, enabling correlation. The body is optional — no `MLVExecutionDefinitionId` is required for a SQL-created MLV that is already catalog-registered.

**Polling status:**

```
GET /api/flt-proxy/liveTableSchedule/getDAGExecStatus/{iterationId}
```

Terminal states: `completed` · `failed` · `cancelled`. Intermediate: `running`. A run that starts must terminate — a hang is an invariant violation.

**Node-level outputs** (the real semantic oracles):  
- `node.warnings` — typed `NodeWarning` values, e.g. `CDFDisabled` (source table lacks Change Data Feed; MLV falls back to full refresh). A CDF change leaves output rows identical — only `node.warnings` shows it.  
- `refresh_policy` — `Incremental` or `Full`; reports which path the node took.  
- `NodeExecutionMetrics` — `added`/`dropped` row counts, `status`, `error_code`/`error_source`.  
- Insights tables: `sys_node_metrics`, `sys_run_metrics` — queryable structured output persisted by FLT per run.

## 3. The Iteration ID

Every `runDAG` call takes an `iterationId` the skill generates (`str(uuid.uuid4())`). This GUID is the **correlation anchor** for the run:

- EDOG interceptors tag every emitted event with this ID.
- `getDAGExecStatus` is keyed on it.
- The verdict cites interceptor events using it.

Generate a fresh GUID for each run. Do not reuse IDs across scenarios.

## 4. Token Lifecycle

### Bearer token (~1 hour)

- Expires ~1 hour from mint.  
- **5-minute refresh buffer**: check `bearerExpiresIn` from `GET /api/edog/health` before any long operation. If under 5 minutes, do not start the operation — re-auth is required.  
- **Auto-refreshes mid-run** only if a username/session is saved in EDOG config. Without a saved session, expiry is fatal.  
- `GET /api/edog/health` fields: `bearerExpiresIn` (seconds), `tokenExpired` (bool), `mwcToken` (state string).  
- A 401/403 mid-run → re-auth required (not retriable without intervention).

### MWC token (Microsoft Work Credentials, ~15 minutes)

- **15-minute refresh buffer**: EDOG refreshes automatically when the buffer is entered.  
- `GET /api/flt/config` reports `bearerToken`/MWC availability.

### Capacity routing

- A 404 with `capacity_routing_not_ready` is a **retriable infra condition**, not a verdict on the change.

## 5. The 11 Interceptors

EDOG instruments the FLT process with 11 interceptors that emit structured events in real time. They are the skill's primary evidence source.

| Interceptor | What it captures |
|-------------|-----------------|
| `log` | FLT structured log lines, severity, CodeMarker context |
| `telemetry` | Telemetry events, correlation IDs, dimensions |
| `http` | Outbound HTTP calls — method, path, status, duration |
| `retry` | Retry attempts — attempt number, delay, trigger cause |
| `dag` | DAG node transitions — nodeId, state, timing |
| `token` | Token mint events — type (bearer/MWC), timestamp, lifetime |
| `spark` | Spark session events — create, query, close, pool state |
| `fileop` | File I/O operations — path, operation type, duration |
| `cache` | Cache hits/misses, evictions |
| `catalog` | Catalog lookups — MLV/table registrations, resolutions |
| `flt-ops` | FLT internal operations — DI resolutions, config reads |

**Critical constraint:** interceptors capture **status, duration, errors, and counts** — they do **NOT** capture method return values. Every grounded claim must cite only fields the interceptors actually emit. "The method returned X" is never supportable from interceptor evidence.

Each interceptor event carries a monotonic `TopicEvent.SequenceId` (per topic) and payload IDs (`IterationId`, `CorrelationId`, `RootActivityId`, `dagId`, `nodeId`, `sessionTrackingId`). In Phase 1 the skill reads interceptor output via REST (`/api/logs`, `/api/telemetry`, `/api/executions`, `/api/edog/interceptors-status`); the full topic-event stream with stable sequence IDs is exposed over REST in Phase 3 via the trace-bundle endpoint.

## 6. Deploy Lifecycle

Deploy is **config-driven**. EDOG patches `flt_repo_path` in the config in place — there is no `--repo-path` argument and no checkout performed by the deploy command itself.

**Skill worktree protocol:** create a git worktree at the PR commit → repoint `flt_repo_path` in `edog-config.json` to the worktree path (record the old value in the teardown ledger as `config_restore`) → deploy → restore `flt_repo_path` → remove worktree on cleanup.

**Triggering deploy:**

```
POST /api/command/deploy
```

**Monitoring via SSE:**

```
GET /api/command/deploy-stream
```

Completion signal: `event: complete` with phase field `running` and `deployMessage: "Deploy complete"`.

Duration: typically ~4 minutes. The skill fires deploy, ends the turn, persists run state, and resumes on the next checkpoint poll.

## 7. Capacity Routing and Infra

FLT requires a Fabric **capacity** to run. The skill provisions or reuses:

1. A Fabric **workspace** (logical container).  
2. Capacity assignment via `/api/fabric/workspaces/{id}/assignToCapacity`.  
3. A **schema-enabled lakehouse** — must pass the Fabric schema flag at creation time. Default lakehouse creation is non-schema, and MLVs using silver-schema syntax (`silver.<name>`) require schemas. This cannot be patched after creation.  
4. A **notebook artifact** — the Jupyter/Spark session is artifact-bound. No notebook artifact → no session.

SQL-created MLVs (`CREATE MATERIALIZED LAKE VIEW silver.<n> AS ...` executed in a notebook Spark session) are catalog-registered and runnable via `runDAG` with no separate `MLVExecutionDefinitionId`. Verified by FLT domain owner.
